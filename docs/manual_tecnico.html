1. Arquitectura

PWA táctil mobile-first sobre LAMP Vanilla (PHP 8.1+, Vanilla JS, sin frameworks). El navegador habla solo con un proxy PHP del mismo origen (api/omni.php) ⇒ cero CORS. El proxy resuelve sesión (cookie HttpOnly + SameSite=Strict), reenvía vía cURL al API CORE añadiendo Authorization: Bearer y X-Interlocutor-Id, y desenvuelve el sobre canónico.

Navegador (JS)
   │  fetch same-origin  api/omni.php?action=...
   ▼
Proxy PHP  ──cURL──►  API CORE v6.6.0  (https://api.omni.josepan.app/api/v1)
   │  Bearer + X-Interlocutor-Id
   ▼  sobre {status,data,message}  → {ok,data,code}

4. Contrato del API (sobre)

El API CORE responde { status:"success"|"error", data, message, error_code }. El SDK lo normaliza a { ok, data, error, code } y el proxy emite al navegador { ok, data, error, code }. El token expira en 8 h; ante 401 ERR_AUTH el cliente emite omni:session-expired y vuelve al login. Las cantidades se muestran con Metrology.formatQty aplicando pack_size (ej. 4 ud (28 kg)).

7. Workflow de traspaso externo

Estados: SOLICITADO → EN_PICKING → LISTO_DESPACHO → EN_RUTA → PENDIENTE_RECEPCION → CERRADO. El stock se descuenta de origen en route y se incrementa en destino en close.

TransiciónBody
POST /transfers{location_id_origin, location_id_destination, items:[{item_id,item_type,batch_id,quantity_requested}], notes}
PUT /{id}/dispatch{items:[{item_id,batch_id,quantity_dispatched}]}
PUT /{id}/route{route_id}
PUT /{id}/close{items:[{item_id,batch_id,quantity_received}]}

El RBAC por transición y estado previo lo valida el API CORE. El proxy delega (mantiene auth + same-origin). Respuestas con transfer_id y state.

8. RBAC de pantallas

En el login [1003] llama mis_pantallas → /my-screens: SuperAdmin recibe "*"; el resto, la lista filtrada por su rol. El SuperAdmin edita el mapa (pantalla → roles) con el Gestor de Permisos (roles_listar, perms_listar, perms_guardar), restringido en el proxy con requireSuperAdmin(). Claves de pantalla: recepcion, ubicar, picking, transporte, solicitar, recibir, merma, dashboard.

Selección de tienda (interlocutor)

Login en dos fases: (1) action=login valida credenciales y lista /catalog/interlocutors; (2) el usuario elige su sede y action=login_sede re-autentica con ese interlocutor_id. API CORE v6.8 fija el rol del JWT según la sede del login, por eso se re-autentica (no basta el header). Cambiar de sede exige logout + login. El filtrado perimetral usa X-Interlocutor-Id.

En el login [1003] llama mis_pantallas → /my-screens, que (API CORE v6.8) devuelve siempre un array de claves: para el SuperAdmin incluye todas las pantallas + gestor_permisos; para el resto, las asignadas a su rol; vacío si ninguna. El subsistema solo itera el array. Claves de pantalla: recepcion, ubicar, picking, transporte, solicitar, recibir, merma, dashboard, gestor_permisos.

Catálogo de pantallas (SSOT)

El Gestor de Permisos pobla su matriz desde GET /rbac/subsystems/1003/screens (catálogo de claves registradas en el API CORE), intersecado con las pantallas que [1003] sabe renderizar (TILE_META); si el catálogo no responde, usa las locales. Así solo se ofrecen claves válidas (evita 422 al guardar) y una clave nueva (p.ej. dashboard) aparece en cuanto se registra en el API CORE. El alta/baja de claves se hace desde el admin del API CORE o vía curl, no desde [1003].

Los selectores operativos solo muestran SKUs con status: "active" (filtro ?status=active + verificación en cliente). El Panel de traspasos es un informe histórico y muestra la información sin importar el estado del SKU.

Búsqueda de SKU (typeahead)

Con el catálogo estandarizado (~1000 SKUs) los selectores de SKU se sustituyen por un buscador con debounce que consulta GET /catalog/skus?q=&status=active&limit=30. Aplica a recepción, ubicación, merma y solicitud.

Recepción contra OC/albarán de [1002]

La pantalla recepcion lista las OC pendientes (GET /purchasing/orders, se filtran las ya recibidas/almacenadas). Al elegir una se carga su detalle (GET /purchasing/orders/{id}) y por cada línea SKU el operario asocia ubicación, lote (ref + caducidad) y cantidad recibida. Al guardar: por línea POST /inventory/reception (alta de stock con lote inline) y luego PUT /purchasing/orders/{id}/receive con {details:[{detail_id, quantity_received}]} para dejar la OC en estado recibido/almacenado.

Parámetros de implantación (GET /system/params)

Tras el login se consulta GET /system/params y se guardan en state.params. Dos modos: Implantación (inventory_restriction=false, stock_negative_allowed=true): el FEFO usa include_empty=1, acepta lotes con stock 0 y no muestra avisos de stock. Producción (inventory_restriction=true, stock_negative_allowed=false): el FEFO solo trae lotes con stock; si no hay, se bloquea; si el lote no cubre lo pedido, avisa. El cambio de modo es automático al releer los parámetros en cada sesión; no requiere cambios de código.

Payload de traspaso (Traslado Externo)

La solicitud envía movement_type:"Traslado Externo", location_id_origin/destination, interlocutor_id_origin/dest, e items[{item_id,item_type,batch_id,quantity_requested}]. El cierre (/close) añade reception_date (fecha física, se propone hoy) junto a items[{...,quantity_received}]. Las tres cantidades (requested/dispatched/received) permiten rastrear pérdidas.

Solicitud de traspaso — lote por FEFO (con stock en origen)

El API exige batch_id por ítem en POST /inventory/transfers. Para no pedir el lote al operario, al añadir cada ítem el cliente consulta GET /inventory/batches?item_id=&location_id={origen} (origen = bodega del OBRADOR) y adjunta el lote FEFO con stock real en esa ubicación; si el API devuelve quantity_available, avisa cuando no cubre lo pedido. Los ítems se envían como {item_id, item_type, batch_id, quantity_requested}.

Listas de traspaso vacías

Las bandejas de picking, transporte y recepción muestran «No hay traspasos pendientes.» tanto si la respuesta es vacía como si el endpoint aún no responde (p.ej. workflow no migrado), evitando un error visible al operario.

Home ordenado y por color

El hub ordena los accesos por TILE_ORDER y colorea el borde de cada tile según el área/rol responsable (Almacén, Transporte, Tienda, Mermas, Gestión), con leyenda.
